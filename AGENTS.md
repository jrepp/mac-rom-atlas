# mac-rom-atlas Agent Guide

This repository is the consolidation point for Old World Macintosh ROM reverse engineering, SuperMario source correlation, transformed reference material, and MCP/REST/frontend tooling.

## Subway Map

Use this map to orient before editing or analyzing.

```text
Reference Line
  source PDFs (local/ignored) -> pdf-book2md output (local/ignored) -> services/mcp-server/data/knowledge_base.json
  Toolbox/OS trap notes -> services/mcp-server/data/traps.json

ROM Line
  data/roms/ (local/ignored raw ROMs)
    -> services/mcp-server/data/projects/*/project.json
    -> services/mcp-server/data/projects/*/nodes.json
    -> services/mcp-server/data/projects/*/edges.json
    -> atlas/maps/<dataset>/roms.tsv

Disassembly Line
  tools/stackimport (intended tool location)
    -> data/disassembly/ (local/ignored listings)
    -> services/mcp-server/data/projects/* rom_disassembly nodes
    -> atlas/maps/<dataset>/functions.tsv
    -> atlas/maps/<dataset>/xrefs.tsv

Structure Line
  server structure pass + stackimport analysis
    -> function_candidate, pointer_table, data_region, resource_marker nodes
    -> atlas/maps/<dataset>/regions.tsv
    -> atlas/maps/<dataset>/pointer-tables.tsv
    -> atlas/maps/<dataset>/data-regions.tsv
    -> atlas/maps/<dataset>/resources.tsv

Source Overlay Line
  data/sources/SuperMarioProj.1994-02-09 (local/ignored)
    -> source-overlay API
    -> SuperMario function-name candidates
    -> disassembly annotations and notes

Workspace Line
  services/mcp-server/src/index.ts
    -> REST/MCP APIs
    -> services/mcp-server/public/index.html
    -> reverse engineering, resources/data, and project-map workflows

Atlas Publication Line
  services/mcp-server/data/projects/* split JSON
    -> tools/export_atlas_maps.js
    -> atlas/maps/<dataset>/*.tsv + notes/*.md + manifest.yaml
```

## Stable Locations

- `services/mcp-server/` is the active MCP/REST server and frontend.
- `services/mcp-server/data/knowledge_base.json` is transformed 68k documentation.
- `services/mcp-server/data/traps.json` is the Toolbox/OS trap database.
- `services/mcp-server/data/projects/` is persistent project analysis state.
- `atlas/maps/` is the clean, diffable publication format.
- `tools/export_atlas_maps.js` converts split project JSON into TSV/Markdown maps.
- `data/roms/`, `data/disassembly/`, `data/sources/`, `data/reference-pdfs/`, and `data/book-extractions/` are local/ignored data staging areas unless policy changes.

## Working Rules

- Keep raw ROM bytes, source PDFs, full memory-page dumps, generated disassembly listings, `node_modules`, and build outputs out of Git.
- Prefer repo-relative atlas paths in metadata. Do not introduce machine-local paths such as `/Users/...`, `/tmp/...`, or `~/Downloads/...` into committed data.
- Treat function boundaries, source-name overlays, code/data splits, and resource classifications as hypotheses unless control flow, xrefs, or source evidence confirms them.
- Preserve provenance on every generated or analyst-authored claim. Use `source`, `confidence`, `evidence`, `target`, and stable IDs where possible.
- Use split, line-oriented map files for public data. Avoid giant nested JSON as the canonical publication format.
- Keep TSV files sorted by address or stable ID so diffs remain useful.
- When changing MCP server behavior, run `npm run build` in `services/mcp-server`.
- When changing frontend JavaScript in `public/index.html`, parse the embedded script with Node before finishing.
- When changing project JSON or map exports, validate JSON parsing and regenerate maps with `node tools/export_atlas_maps.js`.

## Reverse Engineering Workflow

For an address or function-like region:

1. Locate it in the disassembly workflow.
2. Check whether the current bytes are likely code, data, table, resource content, or mixed.
3. Inspect inbound xrefs to understand who can reach it.
4. Inspect outbound xrefs, branches, calls, memory refs, and traps to understand runtime effects.
5. Check nearby atlas map regions for data islands and resource boundaries.
6. Check SuperMario source overlay candidates, but keep names provisional until evidence is strong.
7. Record durable conclusions as notes or map rows, not only as transient chat observations.
8. Regenerate `atlas/maps/` when persistent project data changes.

## Annotation Guidance

Use these terms consistently:

- `confirmed` means supported by direct control-flow, xref, source, or structural evidence.
- `hypothesis` means plausible but not confirmed.
- `candidate` means generated by a tool or heuristic.
- `mixed` means the region may contain both executable instructions and data/resources.
- `source_overlay` means a name or relationship inferred from source-text matching, not binary identity.

Good notes answer:

- What address/range is this about?
- What evidence supports the claim?
- What remains uncertain?
- What should the next agent inspect?

## Public Map Format

Prefer:

- YAML for manifests and small metadata.
- TSV for maps, regions, functions, xrefs, resources, strings, traps, and labels.
- Markdown with YAML front matter for notes.

Avoid:

- Bulk hexdumps as canonical data.
- Reformatting large JSON files without semantic changes.
- Tool-specific temporary paths in committed output.
